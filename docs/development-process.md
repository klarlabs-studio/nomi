# Development Process

How work happens on Nomi. Plan → decide → build → remember.

This document is a **map**, not a methodology import. Each tool below
is already in use; the doc exists so a new contributor (human or AI)
can see the system whole rather than rediscover it piecewise.

## Layers

```
What                          Where                              Why
─────────────────────────────────────────────────────────────────────
Planning + task state         .roady/                            spec → plan → state, hash-chained
Architectural decisions       docs/adr/                          ADRs, numbered, immutable history
Cross-session memory          ~/.claude/projects/<proj>/memory/  auto-memory, persists across runs
Project agent contract        CLAUDE.md                          loaded every session, kept tight
Context budget                docs/context-budget.md             tracks auto-loaded file sizes
Vendor gotchas                docs/vendor-notes.md               append-only library/tool surprises
Changelog                     CHANGELOG.md                       Keep-a-Changelog format, dated entries
Roadmap detail (per-feature)  .roady/spec.yaml                   source of truth for backlog
Roadmap surface (landing)     docs/index.html #roadmap           public Now/Next/Later cards
```

## Planning — Roady

Roady is the planning + tracking layer. The spec lists features; the
plan decomposes them into tasks; state tracks per-task status; events
are hash-chained for audit.

- **`.roady/spec.yaml`** — feature list. Add via `roady_add_feature`
  (MCP), never by hand-editing. Editing the YAML directly breaks the
  hash chain in `.roady/events.jsonl`.
- **`.roady/plan.json`** — task decomposition with `depends_on`
  edges. Regenerated by `roady_generate_plan` (use with care — it
  resets custom tasks that don't match features).
- **`.roady/state.json`** — per-task status (`pending` →
  `in_progress` → `done` / `blocked`). Move tasks with
  `roady_transition_task`.
- **`.roady/events.jsonl`** — append-only, hash-chained audit. Each
  transition links to the previous event's hash. Tampering is
  detectable.
- **`.roady/policy.yaml`** — `max_wip: 3`, `allow_ai: true`. Keep WIP
  ≤ 3.

**Workflow when picking up work**: check `.roady/state.json` for ready
tasks, claim via `roady_transition_task`, work, transition to done.
Skip the `TodoWrite` tool when working in Nomi — Roady is the source
of truth, TodoWrite would split it.

## Decisions — ADRs

Significant architectural calls land in `docs/adr/NNNN-title.md` as
Architecture Decision Records.

- Numbered sequentially (`0001`, `0002`, …).
- Status: `Proposed` → `Accepted` (or `Rejected` / `Superseded by NNNN`).
- Structure: Context → Decision → Consequences → Alternatives → Open
  questions.
- Existing ADRs:
  - `0001` Plugin Architecture — Plugin is the unit of extensibility,
    declares roles via typed manifest.
  - `0002` Plugin Distribution & Lifecycle.
  - `0003` Plugin Transport Tiers.
  - `0004` Mnemos Integration as a Plugin (revised 2026-05-22).

ADRs are **immutable history** in spirit. Revisions land as a new ADR
that supersedes the old one (or, for small fixes, as an in-place edit
with a dated note at the top). Don't quietly rewrite history.

## Cross-session memory — auto-memory

Claude Code's auto-memory at
`~/.claude/projects/-Users-felixgeelhaar-Developer-projects-open-source-nomi/memory/`
persists facts across sessions. Backed by individual `*.md` files
indexed by `MEMORY.md` (which is auto-loaded every session, capped at
~200 lines).

What to write there:
- **User preferences** — how the user likes to work.
- **Feedback** — corrections + validations from past sessions.
- **Project facts** — non-obvious context, dates, decisions with
  rationale.
- **External references** — pointers to where information lives in
  other systems.

What NOT to write there:
- Anything derivable from the current repo (`git log`, `grep`).
- Anything already in `CLAUDE.md` or ADRs.
- Ephemeral session state — that's what plans + tasks are for.

## Agent contract — CLAUDE.md

The single file every Claude Code session loads. Keep it tight (~150
lines max). Sections in the current project:

- **Project** — what Nomi is, where data lives.
- **Common commands** — the make targets and `go test` invocations
  contributors run hourly.
- **Architecture** — runtime flow (Run → Plan → Steps → Tools), wiring
  order (`cmd/nomid/main.go`), persistence layout.
- **Conventions** — module path, where types live, how to add tools /
  connectors.
- **Roadmap (Roady)** — pointers into `.roady/` + workflow rules.

Avoid `AGENTS.md` or other parallel agent-contract files. CLAUDE.md is
canonical.

## Context budget — `docs/context-budget.md`

Tracks the files that auto-load on every Claude Code session vs the
model's context window. Goal: keep auto-loaded total under ~30% of the
window so room for actual work stays plentiful.

Reviewed quarterly or when a new heavy reference lands. Trim when
auto-loaded total approaches 50K on a 200K-window model.

## Vendor gotchas — `docs/vendor-notes.md`

Append-only. One entry per surprise that someone hit and would hit
again without a note. Examples in the current file:

- Tauri WKWebView SSE limitation on macOS → IPC bridge workaround.
- Vite dev (5173) vs preview (4173) port mismatch.
- SQLite FK pragma off-by-default trap.
- golang-migrate up+down pair requirement.
- shadcn components are vendored, not installed.

Format: `### <Topic> — YYYY-MM`. Short. Point at code or docs for
detail; don't restate.

## Changelog — `CHANGELOG.md`

Keep-a-Changelog format. Each notable change lands as an entry under
`[Unreleased] - YYYY-MM-DD` with a short banner describing the
intent. Sections: Added / Changed / Removed / Deprecated / Fixed /
Security / Roadmap.

Commit reference style: `[Unreleased] - 2026-05-22 (short banner)`.
Entries grouped by banner so a reader can scan one cycle at a time.

## Session-end discipline

At the end of a working session:

1. Are tasks updated in Roady? `roady_transition_task` for anything
   that moved.
2. Did you decide anything new and significant? Capture as an ADR or
   as a `feedback` / `project` memory entry.
3. Did the changelog get an entry for the cycle?
4. Did any vendor surprise hit? Append to `docs/vendor-notes.md`.
5. Did auto-loaded file sizes shift? Update
   `docs/context-budget.md`.

These are not a checklist for every commit — they're a checklist for
when work pauses and the next session needs to pick up cold.

## What this process is not

- **Not Agent OS.** Agent OS proposes a parallel `memory/` directory
  inside the repo with `status.md`, `decisions.md`, `open-threads.md`,
  `sessions/`, etc. That parallel structure was considered and
  rejected: Roady already covers planning + state, ADRs cover
  decisions, auto-memory covers session continuity, CLAUDE.md covers
  the agent contract. Adding a second store of any of those would
  split the source of truth.
- **Not waterfall.** Roady doesn't require up-front design for every
  task. The decomposition step is cheap; the spec evolves as the
  product does.
- **Not a checklist gate.** No process police. The goal is that
  someone reading the repo six months from now can reconstruct what
  was done and why.
